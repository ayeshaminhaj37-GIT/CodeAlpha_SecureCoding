Vulnerabilities Found & Fixes

V1 · SQL Injection
Critical

User input is directly concatenated into SQL queries without sanitization. An attacker can manipulate the query logic to bypass authentication, dump the database, or delete records.

❌ Vulnerable Code
// Direct string concat — DANGEROUS
String query = "SELECT * FROM users"
  + " WHERE username='"
  + username + "'"
  + " AND password='"
  + password + "'";
// Attack: username = "admin'--"
// Bypasses password check entirely
✅ Secure Fix
// Parameterized query — SAFE
String query = "SELECT password_hash"
  + " FROM users"
  + " WHERE username = ?";
PreparedStatement ps =
  conn.prepareStatement(query);
ps.setString(1, username);
// Input treated as data, not code
ImpactAuthentication bypass, full database dump, data deletion (OWASP A03:2021)
Fix AppliedUse PreparedStatement with parameterized queries for all database operations.
V2 · Hardcoded Credentials
Critical

Database credentials and admin passwords are hardcoded in the source code. Anyone with source access — including version control history — can extract them permanently.

❌ Vulnerable Code
static final String DB_PASSWORD
  = "admin123";
static final String ADMIN_PASS
  = "password";
// Visible in source & git history
✅ Secure Fix
// Load from environment variables
static final String DB_PASSWORD
  = System.getenv("DB_PASSWORD");
// Or use a secrets manager
// (AWS Secrets Manager, Vault)
ImpactFull database and system compromise if code is leaked or committed to public repositories.
Fix AppliedUse environment variables or a secrets manager. Never commit credentials to version control.
V3 · Weak Password Hashing (MD5)
Critical

MD5 is cryptographically broken. It has known collision vulnerabilities, no salting, and rainbow tables crack MD5 hashes in seconds. Passwords stored with MD5 are effectively plain text.

❌ Vulnerable Code
MessageDigest md =
  MessageDigest.getInstance("MD5");
// MD5 is broken — cracked in seconds
// No salt — rainbow table attack
return password; // plain text on fail!
✅ Secure Fix
// PBKDF2 + unique salt per user
PBEKeySpec spec = new PBEKeySpec(
  password.toCharArray(),
  salt.getBytes(), 310000, 256
);
SecretKeyFactory factory =
  SecretKeyFactory.getInstance(
    "PBKDF2WithHmacSHA256"
  );
ImpactIf database is breached, all user passwords are instantly recoverable via rainbow tables.
Fix AppliedPBKDF2WithHmacSHA256 with 310,000 iterations and unique cryptographic salt per user.
V4 · Missing Input Validation
High

Transfer amounts are not validated — negative amounts can drain accounts. Account numbers have no format checks, and no balance verification is performed before deducting funds.

❌ Vulnerable Code
public void transferMoney(
  String from, String to, double amount) {
  // No validation at all!
  // amount = -5000 steals money
  stmt.executeUpdate(debit);
  stmt.executeUpdate(credit);
  // No transaction — money lost on fail
}
✅ Secure Fix
if (amount <= 0) return false;
if (!from.matches("[A-Z0-9]{10}"))
  return false;
conn.setAutoCommit(false);
// Check balance before deducting
if (balance < amount) rollback();
conn.commit();
ImpactAttacker transfers negative amounts to steal funds. Partial failure leaves accounts in inconsistent state.
Fix AppliedValidate amount, account format, balance. Wrap both operations in atomic database transaction.
V5 · Sensitive Data in Logs
High

Full credit card numbers and PII are written to plain text log files. Log files are rarely secured and are often accessible to developers, log aggregators, and attackers who gain file access.

❌ Vulnerable Code
System.out.println("Card="
  + cardNumber);  // Full 16 digits!
fw.write("User: " + user
  + " Card: " + cardNumber);
// Written to plain text file
✅ Secure Fix
// Mask card — show last 4 only
String masked = "****-****-****-"
  + card.substring(card.length()-4);
logger.info("Card=" + masked);
// UserID hashed, not plain name
logger.info("ID="+user.hashCode());
ImpactPCI-DSS violation. Full card numbers in logs can be stolen and used for fraud.
Fix AppliedMask card numbers, hash user identifiers, use structured logger. Never log raw PII or financial data.
V6 · Insecure Session Token
High

Session tokens are generated using predictable data (username + timestamp). An attacker who knows the username can brute-force the timestamp and hijack any active session.

❌ Vulnerable Code
return username
  + System.currentTimeMillis();
// Predictable! Easily guessable.
// "admin1716812345678"
✅ Secure Fix
SecureRandom random =
  new SecureRandom();
byte[] tokenBytes = new byte[32];
random.nextBytes(tokenBytes);
return Base64.getUrlEncoder()
  .encodeToString(tokenBytes);
// 256-bit cryptographic random token
ImpactSession hijacking — attacker takes over authenticated user sessions without knowing the password.
Fix AppliedUse SecureRandom to generate 256-bit cryptographically unpredictable session tokens.
V7 · Path Traversal (File Access)
Medium

User-supplied filename is directly appended to a base directory path without sanitization. An attacker can use "../" sequences to escape the intended directory and read any system file.

❌ Vulnerable Code
File file = new File(
  "/statements/" + filename);
// Attack: filename="../etc/passwd"
// Reads /etc/passwd !
✅ Secure Fix
String safe = filename
  .replaceAll("[^a-zA-Z0-9._-]","");
File base = new File(
  "/statements").getCanonicalFile();
File file = new File(base, safe)
  .getCanonicalFile();
if(!file.getPath()
  .startsWith(base.getPath()))
  return; // Block traversal
ImpactAttacker reads /etc/passwd, private keys, config files, and any file the server process can access.
Fix AppliedSanitize filename, use canonical path comparison to ensure file stays within allowed directory.
V8 · Missing Authorization Check (IDOR)
Critical

The application fetches account details based purely on the account number provided by the user, with no verification that the logged-in user actually owns that account. This is an Insecure Direct Object Reference (IDOR).

❌ Vulnerable Code
public void viewAccountDetails(
  String accountNo) {
  // No ownership check!
  // Any user sees any account
  query = "SELECT * FROM accounts"
    + " WHERE account_no='"
    + accountNo + "'";
}
✅ Secure Fix
public void viewAccountDetails(
  String loggedInUser,
  String accountNo) {
  if(!userOwnsAccount(
      loggedInUser, accountNo))
    return; // Deny silently
  // Then fetch with PreparedStatement
}
ImpactAny authenticated user can view balances, transactions, and details of every account in the database.
Fix AppliedAlways verify the logged-in user owns the requested resource before returning any data.

Recommendations & Best Practices

🛡️
Always use PreparedStatement for database queries

Never concatenate user input directly into SQL. Parameterized queries make injection structurally impossible.

🔐
Store credentials in environment variables or a secrets manager

Use AWS Secrets Manager, HashiCorp Vault, or .env files excluded from version control.

🔑
Use PBKDF2, bcrypt, or Argon2 for password hashing

Never use MD5, SHA1, or SHA256 directly for passwords. Always add unique per-user salts.

✅
Validate and sanitize all user inputs

Check type, range, format, and length before processing. Reject anything that doesn't conform.

📋
Never log sensitive data — mask PII and financial info

Mask card numbers (show last 4), hash user IDs, and never write passwords to any log.

🔒
Always verify authorization before serving any resource

Check that the logged-in user owns or has permission to access every object they request.

⚡
Use database transactions for multi-step financial operations

Wrap debit + credit in a single atomic transaction. Roll back completely on any failure.

🔍
Run static analysis tools regularly

Use SpotBugs, SonarQube, or OWASP Dependency Check in your CI/CD pipeline.
